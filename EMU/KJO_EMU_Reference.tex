\section{Valve Control Subsystem}
%═══════════════════════════════════════════════════════════════════════════════

\subsection{Valve Base Class (\sw{KJO\_Valve})}

The \sw{Valve} class encapsulates a servo-driven, potentiometer-feedback valve:

\begin{itemize}[nosep]
  \item \textbf{Drive}: PWM signal to PCA9685 servo wing channel.
  \item \textbf{Feedback}: Potentiometer on ADS1015 ADC channel; position
        expressed as 0--100\% open.
  \item \textbf{Position control}: \sw{openTo(\%)} sets target; \sw{update()}
        drives servo until pot reading reaches target within dead-band.
  \item \textbf{State queries}: \sw{isMoving()}, \sw{isStopped()},
        \sw{isOpen()}, \sw{isClosed()}.
\end{itemize}

\subsection{Deferred Valve State Machine}

Valve movements are not blocking.  \sw{Check\_Valve\_Move()} is called every
loop iteration and monitors the active valve:

\begin{figure}[H]
\centering
\begin{tikzpicture}[node distance=2.5cm]
  \node[statenode] (idle)   {IDLE};
  \node[statenode, right=of idle] (moving) {MOVING};
  \draw[statetransition, bend left=20]
    (idle) edge node[above, font=\tiny, align=center]
      {valve command\\received}
    (moving);
  \draw[statetransition, bend left=20]
    (moving) edge node[below, font=\tiny, align=center]
      {pos. reached\\or timeout}
    (idle);
  \draw[statetransition, loop above, looseness=5]
    (moving) edge node[above, font=\tiny] {still moving} (moving);
\end{tikzpicture}
\caption{Deferred valve state machine.}
\end{figure}

A watchdog timer prevents the system from remaining in MOVING indefinitely
if the servo stalls or the potentiometer fails.

\subsection{Master\_Valve Class (\sw{KJO\_Master\_Valve})}

\sw{Master\_Valve} extends \sw{Valve} with the CMD/STATUS GPIO handshake for
the Fill valve.  On \sw{open()} or \sw{close()}, the class asserts
\sig{AUX\_IO\_1} (CMD) and then polls \sig{AUX\_IO\_2} (STATUS) to confirm that
the GSEMU has completed the commanded movement.  The GSEMU drives the servo;
the EMU's \sw{Valve} base class is not used for Fill valve servo drive.

%═══════════════════════════════════════════════════════════════════════════════
\section{Fill Valve Handshake Protocol}
%═══════════════════════════════════════════════════════════════════════════════

The Fill valve servo and potentiometer reside on the GSEMU.
The EMU and GSEMU exchange state using two GPIO lines:

\begin{center}
\begin{tabular}{@{}lll@{}}
\toprule
Signal & Driver & Meaning when HIGH \\
\midrule
\sig{AUX\_IO\_1} (CMD)    & EMU   & Fill valve commanded open \\
\sig{AUX\_IO\_2} (STATUS) & GSEMU & Fill valve has reached commanded position \\
\bottomrule
\end{tabular}
\end{center}

\textbf{Open sequence:}
\begin{enumerate}[nosep]
  \item EMU calls \sw{Fill\_Valve.open()}.
  \item \sw{Master\_Valve} asserts CMD = HIGH.
  \item GSEMU \sw{Slave\_Valve.update()} detects CMD rising edge; drives servo to open.
  \item GSEMU asserts STATUS = HIGH when servo reaches open position.
  \item EMU reads STATUS = HIGH; reports completion.
\end{enumerate}

\textbf{Close sequence}: CMD de-asserted (LOW); GSEMU closes servo; STATUS driven LOW.

%═══════════════════════════════════════════════════════════════════════════════
\section{Command Dispatch (\sw{Execute\_Command})}
%═══════════════════════════════════════════════════════════════════════════════

\sw{Check\_Radio()} receives and parses packets from the RCU.
\cmd{HEARTBEAT} is handled directly and a \texttt{Heartbeat\_Response\_Packet}
is returned immediately, containing EMU voltage and current servo positions.
All other commands are dispatched through \sw{Execute\_Command(command\_ID, param)},
which implements a \texttt{switch} on command code.  Commands with deferred
completion (e.g., \cmd{DUMP}, \cmd{BEGIN\_FILL}) put the valve state machine
into MOVING state; the acknowledgement is sent when the movement completes in
\sw{Check\_Valve\_Move()}.

\cmd{QR\_RELEASE} (code~25) is dispatched directly through
\sw{Execute\_Command()} to \sw{QR\_Master::commandRelease()}, which issues a
\sw{QR\_RELEASE\_PULSE\_MS}~(100\,ms) blocking HIGH pulse on \sig{AUX\_IO\_3}.
The response is returned immediately after the pulse; no deferred state machine
is involved.  The call is idempotent: subsequent \cmd{QR\_RELEASE} commands are
no-ops once the servo has been released.

%═══════════════════════════════════════════════════════════════════════════════
\section{Umbilical Quick-Release (\sw{QR\_Master})}
%═══════════════════════════════════════════════════════════════════════════════

\subsection{Overview}

\sw{QR\_Master} is the EMU-side proxy for the umbilical quick-release actuator
that resides on the GSEMU.  It drives \sig{AUX\_IO\_3} to command the GSEMU
\sw{QR\_Slave} state machine.

\subsection{Signal Protocol}

\begin{center}
\begin{tabular}{@{}lll@{}}
\toprule
Signal & Driver & State \\
\midrule
\sig{AUX\_IO\_3} LOW  & EMU & System armed; GSEMU servo held at hold position \\
\sig{AUX\_IO\_3} HIGH (100\,ms pulse) & EMU & Release commanded \\
\bottomrule
\end{tabular}
\end{center}

\sw{QR\_Master::begin()} drives \sig{AUX\_IO\_3} LOW immediately on startup,
ensuring the GSEMU servo is held in the armed position until an explicit
\cmd{QR\_RELEASE} command is received.

\subsection{QR\_Master API}

\begin{center}
\begin{tabular}{@{}ll@{}}
\toprule
Method & Action \\
\midrule
\sw{begin()} & Configures pin as OUTPUT, drives LOW (armed) \\
\sw{commandRelease()} & Pulses HIGH for \sw{QR\_RELEASE\_PULSE\_MS}, then drives LOW \\
\sw{isReleased()} & Returns true after first call to \sw{commandRelease()} \\
\bottomrule
\end{tabular}
\end{center}

\begin{warningbox}
\cmd{QR\_RELEASE} cannot be undone.  Once issued, the umbilical is physically
disconnected.  Ensure the launch sequence has been verified before sending this
command.
\end{warningbox}

%═══════════════════════════════════════════════════════════════════════════════
\section{SD Logging}
%═══════════════════════════════════════════════════════════════════════════════

Log files are stored in \texttt{EMU\_Logs/Log\_N.txt}.  Each record is
time-stamped (PCF8523) and records command receipt, valve movements, sensor
readings, and fault events.
